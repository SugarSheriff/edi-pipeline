Reference

What each segment is actually saying

X12 reads like shorthand until you've stared at enough of it. Here's the vocabulary this demo leans on.

ISA

Interchange envelope

The outermost wrapper — sender ID, receiver ID, control number. Get this wrong and the whole file is unroutable, not just unreadable.

GS / GE

Functional group

Groups same-type transactions together (all 850s in one group). The control numbers here have to match on close, or your ack rejects the whole group.

ST / SE

Transaction set

One actual document — one purchase order. SE's segment count has to match reality; it's the checksum nobody notices until it's wrong.

BEG

Beginning segment

PO number, order type, date. Miss this and there's no anchor to reconcile against when the vendor calls asking "did you get order 4471?"

PO1

Line item

Quantity, unit, price, product ID. Repeats once per line — this is where most real-world mapping bugs live, because the loop is easy to get almost right.

N1 / N3 / N4

Party & address

Who it's for and where it ships. Optional in the spec, load-bearing in practice — ship-to errors are the ones warehouses call about.

Under the tape

The building blocks, for real

The demo above is a simulation. These are the actual pieces you'd wire into a production pipeline — a segment reader, a typed contract for the output, and the job that watches it run unattended.

Splits on segment/element delimiters read from ISA itself rather than hardcoded — trading partners don't all use ~ and *, and assuming they do is how you get silent truncation on day one.

public sealed class X12Reader
{
    private readonly char _segTerm;
    private readonly char _elemSep;

    public X12Reader(string rawDocument)
    {
        // ISA is fixed-width; delimiters live at known offsets,
        // not guessed from config that will eventually drift.
        _elemSep = rawDocument[3];
        _segTerm = rawDocument[105];
    }

    public IEnumerable<Segment> Read(string rawDocument)
    {
        foreach (var raw in rawDocument.Split(_segTerm,
                     StringSplitOptions.RemoveEmptyEntries))
        {
            var elements = raw.Split(_elemSep);
            yield return new Segment(elements[0], elements[1..]);
        }
    }
}

public readonly record struct Segment(string Id, string[] Elements)
{
    public string this[int i] => i < Elements.Length ? Elements[i] : string.Empty;
}

The contract downstream systems actually consume. Required fields are non-optional on purpose — a mapper that "just" marks everything optional pushes the missing-field bug three services further down, where it's much harder to trace back.

export interface Edi850PurchaseOrder {
  poNumber: string;
  poDate: string;
  orderType: "SA" | "RL" | "NE";
  shipTo: Address;
  lines: PurchaseOrderLine[];
}

export interface PurchaseOrderLine {
  lineNumber: number;
  quantity: number;
  unitOfMeasure: string;
  unitPrice: number;
  productId: string;
}

export interface Address {
  name: string;
  street: string;
  city: string;
  state: string;
  postalCode: string;
}

export function validate(po: Partial<Edi850PurchaseOrder>): string[] {
  const errors: string[] = [];
  if (!po.poNumber) errors.push("BEG02: missing PO number");
  if (!po.lines?.length) errors.push("PO1: no line items found");
  po.lines?.forEach((l, i) => {
    if (!l.productId) errors.push(`PO1 loop ${i + 1}: missing product ID`);
  });
  return errors;
}

The unglamorous half of any EDI pipeline: nobody's watching the inbox folder at 2am, so the script has to notice a stuck file itself and say so loudly.

param(
    [string]$InboxPath = "\\edi-server\inbound\850",
    [int]$StaleMinutes = 15
)

$stuck = Get-ChildItem -Path $InboxPath -Filter *.edi |
    Where-Object { $_.LastWriteTime -lt (Get-Date).AddMinutes(-$StaleMinutes) }

if ($stuck) {
    foreach ($file in $stuck) {
        Write-Warning "Stuck file: $($file.Name), age $([math]::Round(((Get-Date) - $file.LastWriteTime).TotalMinutes)) min"
    }
    Send-MailMessage -To "integrations-oncall@company.com" `
        -Subject "EDI inbox has $($stuck.Count) stuck file(s)" `
        -Body ($stuck.Name -join "`n") `
        -SmtpServer "smtp.company.com" -From "edi-monitor@company.com"
    exit 1
}

Write-Output "Inbox clear — $(Get-Date -Format u)"
exit 0
Why this exists

The unglamorous middle of integration work

Most integration demos show the happy path between two shiny APIs. EDI is the opposite end of the spectrum — 40-year-old fixed-width formats, delimiters that vary by trading partner, and validation that has to happen before a single field reaches your ERP. It's less visually exciting than a webhook diagram, and it's most of what actually keeps supply chains running. This repo is for that half of the job.
